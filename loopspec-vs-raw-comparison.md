### Side-by-side: Schema Comparison

**Raw AI Schema (for this idea):**
```sql
CREATE TABLE products (
  id INT PRIMARY KEY,                       -- should be UUID
  artist_id INT,                            -- no FK constraint
  name VARCHAR(255),                        -- mixed naming
  price DECIMAL(10,2),
  image_url TEXT,
  created_at TIMESTAMP
);
-- No RLS, no indexes, no status column
```

**LoopSpec Schema (deterministic):**
```sql
CREATE TABLE products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  seller_id UUID NOT NULL REFERENCES users(id),  -- FK explicitly
  title TEXT NOT NULL,                            -- consistent TEXT
  price NUMERIC(10,2) NOT NULL,                  -- proper money type
  stock INTEGER NOT NULL DEFAULT 0,
  status TEXT CHECK (status IN ('active','draft','archived')),
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_products_seller ON products(seller_id);
-- RLS: Public read. Seller can write own.
```

## Part 5: The 10 Things That Make LoopSpec Better

### 🔥 1. Deterministic Database Schemas (The Biggest Win)

**Problem:** AI models hallucinate database schemas. They use wrong types (FLOAT for money), miss foreign keys, skip indexes, forget RLS policies.

**LoopSpec fix:** The Schema.md template uses industry-aware `generateCoreModels()` that produces **correct, production-ready table definitions** every time. For ecommerce: `NUMERIC(12,2)` for money, UUID PKs, explicit FK constraints, indexes on foreign keys, RLS policies. **Zero hallucinations.**

### 🔥 2. All UI States Enforced

**Problem:** AIs skip edge cases. They build the "happy path" and forget loading, error, empty states.

**LoopSpec fix:** Every screen in AppFlow.md lists ALL required states:
```
| Product Catalog | /products | public | loading, empty, data, filtered |
```
The AI instructions explicitly say: "Handle ALL listed states (loading, empty, error, data)." Then **drift detection** catches if the implemented code is missing any of these.

### 🔥 3. Stack-Specific Anti-Patterns

**Problem:** An AI will happily use patterns that work against your chosen stack.

**LoopSpec fix:** When you pick `nextjs-supabase-shadcn`, LoopSpec injects 7 stack-specific anti-patterns:
- "Never use getServerSideProps (use Server Components)"
- "Never access Supabase client-side without RLS"
- "Never skip loading/error states in async components"

These come from the stack preset's `anti_patterns` field — curated for each of the 16 supported stacks.

### 🔥 4. Token-Efficient Context Routing

**Problem:** Raw "dump everything" context costs 8-15K tokens per task.

**LoopSpec fix:** The context router classifies tasks by domain (frontend/backend/fullstack/design/devops) and only routes relevant spec sections. A frontend task doesn't get the Schema.md. Token budget = default 15K, configurable. Saves **~60-80% token waste** per task.

### 🔥 5. Automated Drift Detection

**Problem:** Code drifts from spec silently. You don't notice until it breaks.

**LoopSpec fix:** `loopspec_watch` scans recently modified files and checks 7 categories of drift (auth, UI states, validation, type-safety, conventions, API contract, route drift). Each check returns a severity-ranked report with suggested fixes.

### 🔥 6. Bayesian Pattern Learning

**Problem:** Every task starts from zero knowledge.

**LoopSpec fix:** After each task, `loopspec_compound` extracts learnings, stores them with Bayesian confidence scores (α/β from Beta distribution), and promotes high-confidence patterns to SKILL.md and the cross-project playbook. **The system literally gets smarter over time.**

### 🔥 7. Guardrails (Prevent Before Happening)

**Problem:** Security, accessibility, and performance are afterthoughts.

**LoopSpec fix:** `loopspec_preflight` injects 40 rules from 4 packs (OWASP, WCAG 2.1 AA, React patterns, performance budget) **before** the AI starts coding. The AI gets told: "Validate all user inputs", "4.5:1 contrast ratio", "LCP < 2.5s" — upfront.

### 🔥 8. Scorecard (Objective Quality Measurement)

**Problem:** "Is this good code?" is subjective.

**LoopSpec fix:** The scorecard analyzes **6 real dimensions**:
1. Spec compliance — types defined, error handling, validation, TODOs
2. Pattern match — TypeScript strictness, `@ts-ignore` usage
3. Drift score — results from drift detection
4. Test coverage — file-level test presence
5. Accessibility — alt text, labels, aria, div onClick
6. Design match — hardcoded colors, inline styles, magic pixels

Each dimension scored 0-100 with detailed breakdown. Persisted to SQLite for trend tracking.

### 🔥 9. Enforcement (CI/CD Integration)

**Problem:** Specs are nice. Enforcing them is hard.

**LoopSpec fix:** `loopspec_enforce` generates:
- **Pre-commit hooks** that block commits with high-severity drift
- **GitHub Actions** that fail PRs below a configurable score threshold (default 70/100)
- **GitLab CI** configs
- **PR comments** with auto-generated scorecard

Your spec is now **enforced, not just documented**.

### 🔥 10. Cross-Project Learning

**Problem:** Learnings from Project A don't help Project B.

**LoopSpec fix:** Patterns with confidence ≥ 0.7 get promoted to a **cross-project playbook** at `~/.loopspec/playbook/index.json`. The `loopspec_infer` tool combines:
- Industry best practices (curated knowledge for fintech, healthcare, ecommerce, etc.)
- Stack-specific patterns (from 16 presets)
- Your global playbook (patterns you learned)
- This project's Bayesian top patterns
